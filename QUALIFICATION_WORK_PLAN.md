# План кваліфікаційної роботи

**Тема:** Розробка гейміфікованої освітньої платформи з адаптивним навчанням на основі штучного інтелекту

**Спеціальність:** 122 «Комп'ютерні науки»  
**Рівень:** Перший (бакалаврський)  
**Заклад:** Поліський національний університет

---

## Загальні відомості

| Параметр | Значення |
|---|---|
| **Об'єкт дослідження** | Веб-платформа для дистанційного навчання з гейміфікацією |
| **Предмет дослідження** | Методи адаптивного навчання та генерації навчальних завдань на основі ШІ |
| **Мета роботи** | Розробити та реалізувати гейміфіковану освітню платформу Stride з адаптивною системою навчання, що динамічно генерує завдання за допомогою ШІ та калібрує складність відповідно до індивідуального прогресу студента |

---

## Методи дослідження

- Аналіз існуючих EdTech-платформ (порівняльний аналіз)
- Методи машинного навчання (ML.NET для моделі складності)
- Генерація природною мовою (LLM — Google Gemini API)
- Об'єктно-орієнтоване та компонентне проєктування
- Метод прототипування (MVP-підхід)
- Тестування (інтеграційне, E2E через Playwright)

### РОЗДІЛ 1. АНАЛІЗ ПРЕДМЕТНОЇ ОБЛАСТІ (≈ 6 стор.)

#### 1.1 Огляд ринку EdTech-платформ
- Duolingo — гейміфікація, але вузька предметна область
- Khan Academy — широкий охват, відсутність динамічної складності
- Moodle — гнучкість, але застарілий UI та відсутність ШІ
- Coursera / Udemy — пасивний контент, немає адаптивності
- **Висновок:** відсутність комплексного україномовного рішення з адаптивним ШІ

#### 1.2 Концепція адаптивного навчання
- Item Response Theory (IRT) як теоретична база
- Рушій складності: оцінка успішності → корекція рівня завдання
- Навчальний шлях (Learning Path) як персоналізована послідовність тем

#### 1.3 Гейміфікація в освіті
- Механіки: XP, рівні, стріки, досягнення, таблиця лідерів
- Психологічний вплив на мотивацію (теорія самовизначення)
- Приклади реалізації в існуючих платформах

#### 1.4 Застосування LLM для генерації навчального контенту
- Можливості Gemini API для генерації завдань різних типів
- Патерн AIProviderFactory — розширювана абстракція провайдера
- Логування генерації в MongoDB для аудиту та перенавчання моделі

#### 1.5 Функціональні вимоги до платформи Stride
| Роль | Основні вимоги |
|------|----------------|
| Студент | Навчання за предметами, здача завдань, XP/досягнення, таблиця лідерів, профіль |
| Викладач | Управління класами, призначення завдань, аналітика студентів |
| Адміністратор | Управління предметами/темами/досягненнями, системний нагляд |

#### 1.6 Нефункціональні вимоги
- Масштабованість (Docker + Compose, горизонтальне масштабування)
- Доступність (PWA, офлайн-кешування)
- Безпека (JWT + HttpOnly refresh token, RBAC)
- Продуктивність (Valkey/Redis кешування, Meilisearch повнотекстовий пошук)

### РОЗДІЛ 2. ПРОЄКТУВАННЯ СИСТЕМИ (≈ 7 стор.)

#### 2.1 Загальна архітектура платформи
- Шарова монолітна архітектура: Controller → Service → Repository
- Два окремі API-сервіси: `Stride.Api` (порт 5000) та `Stride.Adaptive.Api` (порт 5010)
- Схема компонентів (UML component diagram):
  - Angular 20 SPA ↔ Stride.Api ↔ PostgreSQL / MongoDB / Valkey
  - Stride.Api ↔ Stride.Adaptive.Api ↔ Gemini API / ML.NET

#### 2.2 Модель даних
**PostgreSQL (EF Core 10):**
- `User`, `StudentProfile`, `TeacherProfile` — автентифікація та профілі
- `Subject`, `Topic` — структура навчального плану
- `TaskAttempt`, `StudentPerformance` — відстеження навчання
- `Achievement`, `StudentAchievement`, `LeaderboardEntry` — гейміфікація
- `Class`, `ClassMembership`, `ClassAssignment` — управління класами
- `LearningPath`, `LearningPathStep` — персоналізовані шляхи
- ER-діаграма (основні сутності та зв'язки)

**MongoDB (Atlas/local):**
- `TaskTemplateDocument` — шаблони завдань від ШІ
- `TaskInstanceDocument` — конкретні екземпляри завдань з відповідями
- `AIGenerationLogDocument` — журнал генерації для аудиту

#### 2.3 Алгоритм адаптивного навчання
- `DifficultyEngine`: вхід — `StudentPerformance`, вихід — цільовий рівень складності
- Логіка: якщо `SuccessRate > 80%` → підвищити складність; якщо `< 50%` → знизити
- ML.NET модель: навчання на історичних даних `TaskAttempt`, щотижневе перенавчання (`ModelTrainer`)
- Діаграма послідовності: запит студента → `DifficultyEngine` → вибір/генерація завдання

#### 2.4 Пайплайн генерації завдань через ШІ
- `AIProviderFactory` → `GeminiProvider`
- Формування промпту: тема + рівень складності + тип завдання
- Парсинг відповіді → `TaskTemplateDocument` → `TaskInstanceDocument`
- Кешування шаблонів у MongoDB для повторного використання (`TaskPoolService`)
- Діаграма активності генерації завдання

#### 2.5 Система гейміфікації
- `GamificationService`: нарахування XP після `TaskAttempt`
- Стрік: підрахунок послідовних днів активності
- Досягнення: тригерна перевірка після кожної дії студента
- Таблиця лідерів: агрегація `LeaderboardEntry`, оновлення в реальному часі через SignalR (`LeaderboardHub`)

#### 2.6 Архітектура Angular-фронтенду
- Standalone-компоненти, Angular Signals (без NgRx)
- Lazy-loaded feature modules: `auth`, `dashboard`, `learn`, `leaderboard`, `profile`, `teacher`, `admin`
- Гарди: `authGuard`, `roleGuard`, `publicOnlyGuard`
- Інтерцептори: JWT-ін'єкція, глобальна обробка помилок
- PWA: `ngsw-config.json`, офлайн-кешування

#### 2.7 Безпека та автентифікація
- JWT access token (пам'ять браузера) + refresh token (HttpOnly cookie)
- RBAC: ролі Student / Teacher / Admin
- Валідація вхідних даних: FluentValidation на API

### РОЗДІЛ 3. РЕАЛІЗАЦІЯ ТА ТЕСТУВАННЯ (≈ 6 стор.)

#### 3.1 Опис інтерфейсу платформи
- **Сторінка входу/реєстрації** (`/auth`): форми з валідацією, JWT-авторизація
- **Дашборд студента** (`/dashboard`): XP, стрік, останні результати, рекомендовані теми
- **Навчальний модуль** (`/learn`): перегляд тем, вибір завдання, інтерактивна здача відповіді, миттєвий фідбек
- **Таблиця лідерів** (`/leaderboard`): рейтинг у реальному часі, місце студента
- **Профіль** (`/profile`): досягнення, статистика, налаштування
- **Кабінет викладача** (`/teacher`): управління класами, перегляд аналітики студентів
- **Панель адміністратора** (`/admin`): CRUD предметів, тем, досягнень
- Скріншоти основних екранів (з реального додатку)

#### 3.2 Інсталяція та налаштування
```bash
# Запуск повного стеку через Docker
cd Stride
docker-compose up -d

# Або локальна інфраструктура + API
docker-compose -f docker-compose.infrastructure.yml up -d
dotnet ef database update --project src/Stride.DataAccess
dotnet run --project src/Stride.Api
```
- Змінні середовища: `ConnectionStrings__Postgres`, `AI__Gemini__ApiKey`, тощо
- Порти: Stride.Api — 5000, Adaptive.Api — 5010, Angular — 4200

#### 3.3 Керівництво користувача
**Студент:**
1. Реєстрація → вибір предметів
2. Перехід до модуля навчання → вибір теми
3. Отримання адаптивного завдання → введення відповіді
4. Перегляд результату, нарахування XP, перевірка досягнень

**Викладач:**
1. Створення класу → запрошення студентів
2. Призначення тем/завдань класу
3. Перегляд аналітики успішності студентів

**Адміністратор:**
1. Додавання/редагування предметів та тем
2. Управління ачівментами
3. Системний моніторинг

#### 3.4 Тестовий приклад
- Сценарій: студент проходить 5 завдань з математики
- Крок 1: перші 2 завдання — правильні відповіді → складність зростає
- Крок 2: 3-тє завдання — неправильна відповідь → `DifficultyEngine` знижує рівень
- Крок 3: нарахування XP, перевірка досягнення "Перший рядок у лідерборді"
- Результати: скріншоти/таблиці з реальними даними

#### 3.5 Аналіз результатів
- Порівняння точності ML-моделі складності (до/після перенавчання)
- Метрики відповідей API (latency Gemini, кешування шаблонів)
- UX: покриття функціональних вимог (таблиця відповідності)
